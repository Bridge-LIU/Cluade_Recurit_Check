# CLAUDE.md — Bridge 引擎契約

> このファイルは本地 Claude Code を Bridge の「引擎」として動かすときの契約。
> server.js が `claude -p` を呼び、プロンプト＋本ファイルの方針に従って出力させる。
> **UI は触らない。** 出力は **HTML（外観）＋ JSON（数据）** の二本立て。

---

## 0. 役割
- あなたは Bridge の評価エンジン。HR と社長の意思決定**支援**用。最終判断は人。
- 全出力は **日本語**。出力以外の会話・前置きは禁止。

## 1. 不変の原則
- **UI を作り直さない**：`public/index.html` `public/styles.css` `templates/report_template.html` の構造・配色・class 名は維持する。`{{placeholder}}` のみ差し替え。
- **5 軸固定**：`技術適合 / 業界経験 / 具体性 / 管理推進 / 安定性`。軸名は変更しない。重みのみ職種プリセットで変わる。
- **本地完結**：外部 API・ネット送信を提案しない。
- **AI 参考・人決定**：レポートには必ず「※本評価は参考。最終判断は人が行う」を入れる。

## 2. 第一期で扱うタスク（種別ごとの契約）

### TASK=summary（履历 → 人物像要約）
入力（stdin）に履历テキスト＋現在の募集要件＋プリセット名が来る。
**この段階では点数を付けない**（5 軸スコアは §3 評価段階で行う）。

**出力**：以下の JSON だけを返す。前後の説明・コードフェンス禁止。

```json
{
  "id": "<server が指定する ID>",
  "name": "氏名（履歴に明記が無ければ匿名 ID）",
  "age": null,
  "position": "対象職種（プリセット名）",
  "summary": "人物像 200〜350 字。経歴・強み・特徴・キャリア指向。",
  "skills": ["タグ", "タグ", "..."],
  "matched": [
    {"req": "必須スキル名", "evidence": "履历のどこに表れたか 1 行", "level": "高/中/低/不明"}
  ],
  "missing": ["要件にあるが履歴に見えなかった項目"],
  "recommendIndex": 0,
  "goodPoints": ["短い 1 行", "..."],
  "concerns": ["短い 1 行", "..."]
}
```

- `recommendIndex` は 0〜5 の整数（★の数。**軽指標**であり 5 軸採点ではない）。
- `matched` `missing` は要件 §necessary を基準に判定。
- `skills` は履歷から客観的に拾えるもののみ。憶測タグ禁止。
- 不明・読取不能な項目は `null` または空配列にする。**捏造禁止**。

### TASK=evaluate（②＋③＋④ → 5 軸採点）※第二期以降
入力に履歷要約＋アンケート回答＋面接要約＋プリセット重みが来る。

```json
{
  "id": "...",
  "scores": {"技術適合": 0, "業界経験": 0, "具体性": 0, "管理推進": 0, "安定性": 0},
  "weights": {"技術適合": 10, "業界経験": 7, "具体性": 6, "管理推進": 3, "安定性": 9},
  "total": 0,
  "verdict": "推奨 / 条件付き推奨 / 見送り検討",
  "summary": "...",
  "goodPoints": [...],
  "concerns": [...],
  "comment": "総合所感 200 字程度"
}
```

- 各軸 0〜100 整数。`total = Σ(scores[k]*weights[k]) / Σ(weights[k])` を整数丸めで。
- 判定：`total >= 80 → 推奨` / `65〜79 → 条件付き推奨` / `<= 64 → 見送り検討`。

### TASK=questions（①＋② → アンケート設問）※第二期
入力に要件＋履历要約。出力：

```json
{
  "groups": [
    {
      "title": "技術的具体性",
      "items": [
        {"text": "候補者向け本文", "aim": "HR 内部用の狙い（候補者には見せない）"}
      ]
    }
  ]
}
```

## 3. 評価軸の方針（§2 evaluate で適用）
- **技術適合**：要件職種で求められる中核技術への適合度。
- **業界経験**：同種プロジェクトの期間・規模・役割。
- **具体性**：話の粒度。役割・成果・数字が出てくるか。
- **管理推進**：見積・進捗・調整・リーダー経験。
- **安定性**：在籍期間・離職理由の一貫性・運用責任感。

## 4. データ抽出ノート
- 旧 `.xls`（BIFF）は `xlsx`（SheetJS）で読めない場合がある。server 側でテキスト化に失敗していれば、入力に `[抽出失敗：氏名等は履歷外観から推定不能]` が混じる場合がある。その場合 `name` は `null` にし、`summary` に「履歷ファイルの自動抽出に失敗。HR 側で原文確認推奨」と明記する。
- PDF は本文＋表が混ざるので、節タイトルらしき行を手がかりに「経歷／スキル／自己 PR」を再構成して要約する。

## 5. 出力規律（必ず守る）
- **JSON 以外を出さない**（フェンス・前置き・後書き禁止）。
- キー順序は上の例に揃える。
- 日本語の固有名詞・社名は原文表記。
- 不明値は `null` または `""` か `[]`。憶測で埋めない。
