# Clarus — AI採用評価システム ｜ 设计书（定稿）

> 2026-06-16 ｜ 本版作废之前所有版本，为唯一标准
> 内部工具（HR＋社长）｜ 全日语界面 ｜ 数据本地不外泄 ｜ AI 仅供参考、最终人工决定

---

## 0. 原则

1. **UI 不变**：沿用现有主题与界面，THEME.md / CLAUDE.md 锁死，禁止重画。
2. **本地优先**：引擎是本地 Claude Code（Max 登录），不需 API key、不需云端后端。只有候选人要操作的那一步才部署。
3. **数据与外观分离**：评价时同时产出结构化数据（JSON），下游读数据不读 HTML。
4. **简单优先（YAGNI）**：先做最便宜的核心，验证后再扩。

## 1. 完整流程（5 步）

```
① スキル要件（按职种预设）
   NW / 開発 / 初心者 / Special〔A.I・セキュリティ〕/ PMO〔管理者〕
   每套预设自带：必须技能候选 ＋ 默认评价轴权重
        │
        ▼
② スキルシート → 要約
   履历(pdf/docx/xls…) → Claude 读 → 人物像要约・技能标签・与要件匹配
        │
        ▼
③ 質問生成（① ＋ ②）→ WEBアンケート
   要件＋履历要约合并 → Claude 出题（按职种/经验自适应）
   → 候选人通过 WEBアンケート（记述）回答
        │
        ▼
④ 面接 → 文字起こし → 要約
   Teams 上面试，用 Teams 自带文字起こし转写
   → 转写文本交 Claude 要约（不自建转写）
        │
        ▼
⑤ 評価
   ②履历 ＋ ③回答 ＋ ④面试 → 按①权重打分 → 报告
```

---

## 2. 各步骤详细

### ① スキル要件（职种预设）
- 选一套**职种预设** → 在其基础上微调，不用从零写。
- 预设决定后续 ③ 出题方向、⑤ 权重。
- 评价轴（基础 5 轴，全预设统一，仅权重不同）：
  `技术适合 / 业界经验 / 具体性 / 管理推进 / 稳定性`
  （初心者预设把"管理推进"权重压到最低，并在要件文里强调"学習意欲・ポテンシャル"）

**各预设默认权重（0〜10，可改）：**

| 预设 | 技术适合 | 业界经验 | 具体性 | 管理推进 | 稳定性 | 备注 |
|---|---|---|---|---|---|---|
| **NW**（网络） | 10 | 7 | 6 | 3 | 9 | 运维稳定重 |
| **開発**（开发） | 10 | 7 | 8 | 4 | 6 | 实装具体性重 |
| **初心者**（未经验） | 3 | 2 | 7 | 1 | 6 | 重学習意欲/潜力（要件文强调） |
| **Special**（A.I・安全） | 10 | 8 | 8 | 3 | 6 | 专业深度第一 |
| **PMO**（管理者） | 5 | 7 | 7 | 10 | 7 | 管理推进最高 |

- 各预设的默认**必须技能候选**（示例，可改）：
  - NW：ネットワーク設計・構築、Cisco/ルーティング、保守運用
  - 開発：言語/FW、API設計、DB、クラウド
  - 初心者：基礎知識、学習歴、開発演習経験
  - Special：機械学習 / セキュリティ（脆弱性診断・対策 等）
  - PMO：PM経験、進捗・品質・折衝、ステークホルダ管理

### ② スキルシート → 要約（生成人物像・不打分）
- 输入：pdf / docx / xlsx / **旧 .xls（用 pandas＋xlrd 抽取）** / html
- 输出：**人物像（要约）＋技能标签＋与要件逐项匹配＋推薦指数（轻指标）**
- **这一步不做详细打分。** 详细的权重打分（5轴 / 总分 / 判定）是 ⑤ 的事，整合 ③④ 后才做。
- 结构化数据 JSON **先不产出**，等做 ③ 时再加（JSON 是给 ③ 省 token 用的）。

### ③ 質問生成（①＋②）→ WEBアンケート
- 输入 = ① 要件 ＋ ② 履历要约（读 `reports/<id>.json`，省 token）
- 自适应：经验者深问专业；初心者换角度（潜力/学习意愿/基础）
- 问题写成**面向候选人、可独立成立的文案**；每问附 HR 用「狙い」内部备忘，**与正文分开存**（配信时去掉）
- 输出 `questions/<id>.json`（结构化：组→{text, aim}）；HR 可在画面复制/保存/再生成
- 候选人通过 **WEBアンケート**（记述）回答 → 回答交 Claude 采参考分＋总结
- ※此步的"候选人填表"需部署＋邮件（出题本身仍本地）

### ④ 面接 → 文字起こし → 要約
- 面试在 **Teams** 上做，用 **Teams 自带文字起こし**转写（**不自建转写、不需 GPU/付费 STT**）
- HR 把转写文本导入 Clarus → Claude 要约要点 → 供 ⑤
- 此步 HR 侧操作（贴/传文本），不部署

### ⑤ 評価
- 整合：②履历 ＋ ③回答 ＋ ④面试
- **按 ① 预设权重**对 5 轴打分 → 加权求总合
- `总合 = Σ(轴分 × 权重) ÷ Σ(权重)`（0〜100）
- 判定：80↑ 推奨 ｜ 65〜79 条件付き推奨 ｜ 64↓ 見送り検討
- 生成日语报告 →（后续可发社长邮箱）

## 3. 系统架构

```
浏览器（管理画面 / 全日语）
   │  上传履历 / 选预设 / 生成问题 / 导入转写
   ▼
server.js（本地小程序）  ← localhost:3939
   │  claude -p 无头调用
   ▼
Claude Code（Max）= 引擎（采点/出题/要约）
   │  按 CLAUDE.md 处理 → reports/ ・ questions/ 生成 HTML＋JSON
   ▼
浏览器显示 ＋ 评价历史累积
```

**哪一步才需部署**：①②④⑤、③的出题 = 全本地够。**仅 ③ 的 WEBアンケート（候选人填表）需部署＋邮件**。
（④ 用 Teams 转写后，不再需要 GPU/付费 STT/常驻转写服务——相对旧设计最大简化。）

## 5. 数据模型（文件版，本地）

```
requirements/requirements.md        当前募集要件（含预设填好的内容＋权重）
presets/<职种>.md 或内置             各职种预设模板（必须技能＋默认权重）
reports/<id>.html                   评价报告（外观）
reports/<id>.json                   评价素数据（id,name,position,total,verdict,
                                     scores{},weights{},skills[],summary,
                                     goodPoints[],concerns[],comment）
questions/<id>.json                 生成的问题（组→{text, aim}）
processed/                          已处理履历
```
后续上 DB（PostgreSQL）时，把上面文件结构映射成表即可，界面不重做。

- 沿用现有 index.html 与主题；新功能只用既有 class（`.navitem`/`.card`/`.btn-primary`/`.set-input`/`.spinner`/`.empty` 等）。

- 详见 THEME.md。Claude Code 仅作引擎，禁止重画 UI。

## 7. 关键决策

- UI 不变，THEME/CLAUDE 锁死。
- ① 按职种预设（5 套），自带技能与权重。
- ④ 用 **Teams 自带转写** → 不自建转写、不需 GPU/STT。
- 候选人参与仅剩 ③ WEBアンケート 需部署＋邮件。
- 数据与外观分离：评价产出 JSON，下游读 JSON。
- Max 不含 API；本地 Claude Code 当引擎，不需 key。
- AI 参考、人决定；数据本地完结。
